Every Transatel API call is protected by OAuth 2.0 using the **client credentials** grant. You exchange a `client_id` and `client_secret` — issued by your Transatel account manager — for a short-lived Bearer token, then include that token in the `Authorization` header of every subsequent request. This page walks you through the full flow, explains token lifetime and renewal, lists all available scopes, and covers every error you might encounter.

## Prerequisites

Before you begin, you need a `client_id` and `client_secret` for your account. If you don't have these yet, contact your **Transatel account manager**. Keep your credentials secure — treat them like passwords and never commit them to source control.

<Warning>
  Your `client_secret` is sensitive. Store it in an environment variable or secrets manager, not directly in your code.
</Warning>

## How the client credentials flow works

The client credentials grant is a two-step machine-to-machine flow:

1. Your application sends its `client_id` and `client_secret` to the token endpoint.
2. Transatel returns an `access_token` that you attach to every API request.

There is no user login or redirect involved — the flow is designed for server-side integrations.

## Step-by-step: obtain a token

<Steps>
  <Step title="Build the Basic authorization header">
    Concatenate your `client_id` and `client_secret` with a colon (`:`), then Base64-encode the result. This encoded string goes in the `Authorization` header as `Basic <encoded>`.

    ```bash
    # Example (never hard-code real credentials)
    echo -n "YOUR_CLIENT_ID:YOUR_CLIENT_SECRET" | base64
    # → WU9VUl9DTElFTlRfSUQ6WU9VUl9DTElFTlRfU0VDUkVU
    ```

    <Tip>
      On macOS, use `base64` directly. On Linux, use `base64 -w 0` to suppress line wrapping.
    </Tip>
  </Step>

  <Step title="Send the token request">
    POST to `https://api.transatel.com/authentication/api/token` with:
    - `Authorization: Basic <encoded-credentials>` header
    - `Content-Type: application/x-www-form-urlencoded` header
    - `grant_type=client_credentials` in the request body

    <CodeGroup>
      ```bash cURL
      curl --request POST \
        --url https://api.transatel.com/authentication/api/token \
        --header "Authorization: Basic $(echo -n 'YOUR_CLIENT_ID:YOUR_CLIENT_SECRET' | base64)" \
        --header "Content-Type: application/x-www-form-urlencoded" \
        --data "grant_type=client_credentials"
      ```

      ```python Python
      import requests
      import base64

      client_id = "YOUR_CLIENT_ID"      # or read from os.environ
      client_secret = "YOUR_CLIENT_SECRET"

      credentials = base64.b64encode(
          f"{client_id}:{client_secret}".encode()
      ).decode()

      response = requests.post(
          "https://api.transatel.com/authentication/api/token",
          headers={
              "Authorization": f"Basic {credentials}",
              "Content-Type": "application/x-www-form-urlencoded",
          },
          data={"grant_type": "client_credentials"},
      )
      response.raise_for_status()
      token_data = response.json()
      access_token = token_data["access_token"]
      ```

      ```javascript JavaScript
      const clientId = process.env.TRANSATEL_CLIENT_ID;
      const clientSecret = process.env.TRANSATEL_CLIENT_SECRET;

      const credentials = Buffer.from(`${clientId}:${clientSecret}`).toString("base64");

      const response = await fetch(
        "https://api.transatel.com/authentication/api/token",
        {
          method: "POST",
          headers: {
            Authorization: `Basic ${credentials}`,
            "Content-Type": "application/x-www-form-urlencoded",
          },
          body: "grant_type=client_credentials",
        }
      );

      if (!response.ok) {
        throw new Error(`Token request failed: ${response.status}`);
      }

      const { access_token } = await response.json();
      ```
    </CodeGroup>
  </Step>

  <Step title="Read the token response">
    A `200 OK` response contains the access token and metadata:

    ```json
    {
      "access_token": "eyJ0eXAiOiJKV1QiLCJhbGciOiJFUzI1NiIsImtpZCI6ImQwNjk4OWY2OWUxNzU2NzQ0NTIwMDFiNDdmZjk4MzNmIn0...",
      "expires_in": 3600,
      "scope": "CONNECTIVITY_MANAGEMENT_READ CONNECTIVITY_MANAGEMENT_WRITE",
      "token_type": "Bearer"
    }
    ```

    <ResponseField name="access_token" type="string" required>
      The Bearer token to include in all subsequent API requests.
    </ResponseField>
    <ResponseField name="expires_in" type="integer">
      Token lifetime in seconds. Always `3600` (one hour).
    </ResponseField>
    <ResponseField name="scope" type="string">
      Space-separated list of OAuth scopes granted to your token.
    </ResponseField>
    <ResponseField name="token_type" type="string">
      Always `Bearer`.
    </ResponseField>
  </Step>

  <Step title="Use the token in API requests">
    Add `Authorization: Bearer <access_token>` to the header of every API call:

    <CodeGroup>
      ```bash cURL
      curl --request GET \
        --url "https://api.transatel.com/connectivity-management/subscribers?iccid=89330126000012345678" \
        --header "Authorization: Bearer YOUR_ACCESS_TOKEN"
      ```

      ```python Python
      import requests

      headers = {"Authorization": f"Bearer {access_token}"}

      response = requests.get(
          "https://api.transatel.com/connectivity-management/subscribers",
          headers=headers,
          params={"iccid": "89330126000012345678"},
      )
      print(response.json())
      ```

      ```javascript JavaScript
      const response = await fetch(
        "https://api.transatel.com/connectivity-management/subscribers?iccid=89330126000012345678",
        {
          headers: {
            Authorization: `Bearer ${access_token}`,
          },
        }
      );
      const data = await response.json();
      ```
    </CodeGroup>
  </Step>
</Steps>

## Token lifetime and renewal

Tokens expire after **3 600 seconds** (one hour). The Transatel API does not issue refresh tokens — when your token expires, simply request a new one using the same client credentials flow described above.

<Tip>
  To avoid failed requests at token expiry, track the `expires_in` value and proactively request a new token a few minutes before the current one expires. A safe strategy is to refresh when fewer than 60 seconds remain.
</Tip>

## OAuth scopes

The scopes granted to your token determine which endpoints and operations you can access. Your account manager configures the scopes for your client credentials. The table below lists all available scopes.

| Scope | What it permits |
|---|---|
| `CONNECTIVITY_MANAGEMENT_READ` | Read subscriber and SIM information |
| `CONNECTIVITY_MANAGEMENT_WRITE` | Provision and modify subscribers |
| `NETWORK-ATTACH_HISTORY_READ` | Read network attach history |
| `NETWORK-ATTACH_DETACH_WRITE` | Send network detach commands |
| `NETWORK-SMS_SMS_READ` | Read SMS messages |
| `NETWORK-SMS_SMS_WRITE` | Send on-net SMS messages |
| `NETWORK-SMS_SMS_WRITE_OFF-NET` | Send off-net SMS messages |
| `NETWORK-OTA_SWITCH2PRIVATE` | Trigger OTA switch-to-private operations |
| `NETWORK-DATA-SESSION_SESSION_READ` | Read active data sessions |
| `NETWORK-USAGE_CDR_READ` | Read call detail records (CDRs) |
| `CATALOG_PRODUCTS_READ` | Read catalog products |
| `CATALOG_OPTIONS_READ` | Read catalog options |
| `CATALOG_PRODUCT_MODEL_READ` | Read product models |
| `CATALOG_PRODUCT_MODEL_WRITE` | Create and modify product models |
| `CATALOG_PRODUCT_MODEL_PUBLISH` | Publish product models |

<Note>
  If a request returns `403 Forbidden`, your token does not have the required scope for that operation. Contact your Transatel account manager to request additional scopes.
</Note>

## Error reference

All errors are returned as JSON. The table below covers every error code the token endpoint can return and what you should do.

| HTTP status | `error` value | Cause | Resolution |
|---|---|---|---|
| `400` | `unsupported_grant_type` | The `grant_type` value is not `client_credentials` | Set `grant_type=client_credentials` in your request body |
| `400` | `invalid_request` | Malformed `Authorization` header, missing `grant_type`, or wrong `Content-Type` | Verify the header is `Basic <base64>`, the body contains `grant_type`, and `Content-Type` is `application/x-www-form-urlencoded` |
| `401` | `invalid_client` | Incorrect `client_id` or `client_secret` | Double-check your credentials; contact your account manager if you believe they are correct |
| `500` | `server_error` | Transatel-side error | Retry with exponential back-off; contact support if the error persists |

<Warning>
  A `401 invalid_client` error always means the credentials themselves are wrong — it does not indicate an expired token. Tokens expire silently; the API will return `401` on subsequent calls, not on the token request.
</Warning>
