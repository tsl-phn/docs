title: "Suspend and Reactivate Subscribers"
sidebarTitle: "Suspend / Reactivate"
description: "POST to /suspend or /reactivate to stop or restore a subscriber's network access. Billing fees may still apply during suspension."
---

Suspending a subscriber immediately blocks their access to network services — voice, data, and SMS — without permanently removing the subscription. This is useful when a device is reported lost or stolen, or when you need to pause service temporarily. Reactivating the subscriber restores full network access according to the subscriber's current rate plan and options. Both operations are asynchronous and follow the standard `transactionId` polling pattern.

<Warning>
  **Billing during suspension:** Depending on your rate plan agreement, recurring subscription charges may continue to accrue while a subscriber is suspended. Verify your billing terms before using suspension as a long-term cost-saving measure.
</Warning>

---

## POST /suspend

Suspend a subscriber, blocking all network access until you explicitly reactivate them.

### Endpoint

```
POST /api/subscribers/sim-serial/{sim-serial}/suspend
```

**Required scope:** `CONNECTIVITY_MANAGEMENT_WRITE`

### Path Parameters

<ParamField path="sim-serial" type="string" required>
  The SIM card serial number (ICCID). Must match the pattern `[0-9]{13,20}`.
</ParamField>

### Request Body

<ParamField body="ratePlan" type="string" required>
  The rate plan to apply at the point of suspension. Maximum 50 characters.
</ParamField>

<ParamField body="metadata" type="object">
  A key-value map of custom attributes to record against this event. Supports up to 10 string entries.
</ParamField>

### Response Fields

Returns `HTTP 201 Created` with a `ProvisioningResponse` body.

<ResponseField name="transactionId" type="string">
  UUID identifying this suspend operation. Poll `GET /api/transactions/{transactionId}` for the final result.
</ResponseField>

<ResponseField name="simSerial" type="string">
  The SIM serial number, echoed from the path parameter.
</ResponseField>

<ResponseField name="transactionStatus" type="string">
  Always `PENDING` on a successful 201 response.
</ResponseField>

### Request Example

```bash
curl --request POST \
  --url "https://api.transatel.com/connectivity-management/subscribers/api/subscribers/sim-serial/8933150000000012345/suspend" \
  --header "Authorization: Bearer <token>" \
  --header "Content-Type: application/json" \
  --data '{
    "ratePlan": "DATA_100MB_MONTHLY",
    "metadata": {
      "reason": "lost_device",
      "ticketRef": "SUP-2024-0871"
    }
  }'
```

### Example Response

```json
{
  "transactionId": "b2c4d6e8-1234-4abc-b000-aabbccddeeff",
  "simSerial": "8933150000000012345",
  "transactionStatus": "PENDING"
}
```

## POST /reactivate

Restore network access for a previously suspended subscriber.

### Endpoint

```
POST /api/subscribers/sim-serial/{sim-serial}/reactivate
```

**Required scope:** `CONNECTIVITY_MANAGEMENT_WRITE`

### Path Parameters

<ParamField path="sim-serial" type="string" required>
  The SIM card serial number (ICCID). Must match the pattern `[0-9]{13,20}`.
</ParamField>

### Request Body

<ParamField body="ratePlan" type="string" required>
  The rate plan to restore the subscriber on. Maximum 50 characters.
</ParamField>

<ParamField body="metadata" type="object">
  A key-value map of custom attributes to record against this event. Supports up to 10 string entries.
</ParamField>

### Response Fields

Returns `HTTP 201 Created` with a `ProvisioningResponse` body.

<ResponseField name="transactionId" type="string">
  UUID identifying this reactivation operation. Poll `GET /api/transactions/{transactionId}` for the final result.
</ResponseField>

<ResponseField name="simSerial" type="string">
  The SIM serial number, echoed from the path parameter.
</ResponseField>

<ResponseField name="transactionStatus" type="string">
  Always `PENDING` on a successful 201 response.
</ResponseField>

### Request Example

```bash
curl --request POST \
  --url "https://api.transatel.com/connectivity-management/subscribers/api/subscribers/sim-serial/8933150000000012345/reactivate" \
  --header "Authorization: Bearer <token>" \
  --header "Content-Type: application/json" \
  --data '{
    "ratePlan": "DATA_100MB_MONTHLY",
    "metadata": {
      "reason": "device_recovered",
      "ticketRef": "SUP-2024-0871"
    }
  }'
```

### Example Response

```json
{
  "transactionId": "c3d5e7f9-5678-4def-c111-bbccddeeff00",
  "simSerial": "8933150000000012345",
  "transactionStatus": "PENDING"
}
```

<Note>
  Poll `GET /api/transactions/{transactionId}` until `asyncStatus` is `DONE` or `ERROR`. See the [Transactions reference](/api-reference/connectivity/transactions) for full details.
</Note>

## Error Responses

Both endpoints share the same error codes:

| HTTP Status | Code | Description |
|---|---|---|
| `400` | `BAD_REQUEST` | Missing required fields or invalid parameter format |
| `403` | `INSUFFICIENT_PERMISSION` | Token does not have the `CONNECTIVITY_MANAGEMENT_WRITE` scope |
| `404` | `SUBSCRIBER_NOT_FOUND` | No subscriber found for the provided SIM serial number |
| `409` | `CONFLICT` | Another provisioning request is already pending on this SIM |
| `500` | `INTERNAL_ERROR` | An unexpected server-side error occurred |
| `501` | `NOT_IMPLEMENTED` | The operation is not supported for this account configuration |
